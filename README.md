# StardewLivingNPCs

这是一个把 **AI 对话** 和 **NPC 行为层** 接到《星露谷物语》里的实验项目总仓库。

仓库当前包含两个部分：

- `LivingNPCs/`：我们自己写的 SMAPI companion mod，负责记忆、状态、行为、关系节奏，以及 AI 对游戏世界的受控影响。
- `ValleyTalk/`：基于 dandm1/ValleyTalk 的本地 fork，负责生成 AI 对话，并额外提供流式对话、日志导出、隐藏元数据和第三方上下文注入。

一句话概括当前设计：

> ValleyTalk 负责“NPC 说什么”，LivingNPCs 负责“NPC 记住什么、为什么这样说、以及在安全范围内能做什么”。

## 当前已经完成到什么程度

### 1. AI 对话链路

- 已能通过 ValleyTalk 进行自由输入式 AI 对话。
- 已支持中文 prompt 和中文游戏内输出。
- 已支持真正的流式回复 UI：玩家输入后进入角色立绘对话框，文本会边生成边显示，不再只是先看到“正在思考”。
- 已支持一条回复跨多个原版对话页继续翻页。
- AI 可以决定当前交流是否自然结束；若应该结束，会返回 `endConversation=true`，不再强行给玩家继续选择。
- 每次 AI 对话都会生成隐藏元数据，供 LivingNPCs 使用：
  - `rapportDelta`：这次聊天的额外好感变化。
  - `memories`：值得长期记住的信息。
  - `emotionImpact`：这次交流对 NPC 长期人际情绪造成的影响。
  - `conflicts`：需要被长期记住的人际摩擦。
  - `ambientFollowUp`：稍后可显示的一句头顶气泡后续对白。
  - `behaviorInfluences`：对话后几分钟到几天内继续起作用的行为倾向。
  - `commitments`：双方明确达成的未来约定。
  - `helpRequests` / `helpRequestUpdates`：NPC 主动提出的小忙，以及后续完成 / 拒绝结果。
  - `actions`：最多一个受控世界动作请求。

### 2. NPC 记忆系统

目前已经不是“只记最近一句话”，而是有几层不同用途的记忆：

| 记忆层 | 作用 |
| --- | --- |
| 最近互动记录 | 记录行为、开始对话、送礼、事件互动、NPC 实际执行过的世界动作 |
| 当前状态 | 记录心情、注意度、开放度、当前回应倾向 |
| 长期熟悉度 | 表示 NPC 对玩家“认识多久、熟不熟” |
| 互动节奏 | 记录今天聊了几次、是否连续几天聊天、是否隔了很久才见面 |
| 长期记忆 | 由 AI 从对话里提炼出的事实、偏好、承诺、边界、关系信息 |
| 玩家偏好记忆 | 单独记录玩家喜欢的物品类别、讨厌的物品、生活习惯、价值观和目标 |
| 社区印象 | 记录别的 NPC 亲眼看到或从亲近圈层听来的玩家社交痕迹 |
| 长期约定 | 单独记录再见面、一起去某处、帮忙做某事等未来计划及其履行状态 |
| 主动求助 | 单独记录 NPC 向玩家提出的小忙、截止日和完成状态 |
| 情绪与冲突记忆 | 单独记录 NPC 对玩家的长期情绪、人际摩擦、严重度和恢复过程 |
| 特殊称呼记忆 | 记录玩家请求 NPC 如何称呼自己，以及 NPC 是否接受 |

当前长期记忆逻辑不是只识别固定句式。ValleyTalk 会从整段 AI 对话里返回结构化记忆候选，LivingNPCs 只保存重要度足够高、未来还有价值的内容。若某条记忆描述的是玩家本人，还会额外打上 `playerPreference` 标记，按“喜欢的物品类别 / 讨厌的物品 / 习惯 / 价值观 / 目标”单独存储。为了避免模型偶尔漏掉特殊称呼，LivingNPCs 还额外保留了一个“玩家请求昵称”的确定性兜底。

长期记忆现在还有一层检索与整理系统：

- 存入时会按“记忆类型 + 稳定主题”去重，重复出现的记忆会合并，而不是无限堆叠。
- 每条长期记忆和玩家偏好记忆都会保存主题、标签、重要度、强化次数、最后更新时间、最近被召回时间和召回次数。
- 标签会从 AI 元数据和文本关键词里共同提取，例如农场、矿洞、钓鱼、图书馆、花、咖啡、夜晚等。
- 每次生成 ValleyTalk 上下文前，LivingNPCs 会根据当前位置、时间、天气、最近行为、送礼、事件、待完成求助和待履行约定计算相关性。
- 只有当前真正相关的几条记忆会进入 prompt，且最近刚被使用过的记忆会被轻微降权，避免 NPC 一直反复提同一件事。
- 长期储存仍保留较多信息，但对话召回会更克制：NPC 可以被这些记忆影响语气，也只会在自然时提其中一两点。

### 3. 情绪与冲突系统

NPC 现在除了短时 `Mood`，还会保存一套更长期的人际状态：

- 当前情绪：`Happy`、`Calm`、`Jealous`、`Worried`、`Grateful`、`Disappointed`、`Uneasy`、`Upset`、`Angry`、`Sad`
- 情绪强度：`0-100`
- 冲突记忆：冲突原因、严重度、状态、是否被道歉 / 礼物 / 时间修复
- 关系信任：独立的 `0-100` 数值，决定 NPC 愿意把话说到多深

触发来源目前包括：

- AI 对话元数据里的 `emotionImpact`
- AI 对话元数据里的 `conflicts`
- 玩家送礼结果
- 久别重逢或玩家受伤时的担心
- 浪漫关系中的旁观情境带来的轻度吃醋
- 完成 `help_task` 约定后的感激
- 失约后的失望

当前规则：

- 喜欢 / 最爱的礼物会提升好情绪，并帮助缓和已有冲突。
- 不喜欢 / 讨厌的礼物会造成不自在或不悦，严重时写入冲突记忆。
- AI 若判断玩家确实道歉或修复了关系，可回传 `apology` 与 `repairDelta`。
- `关系信任` 与 `履约信任` 是两套值：前者决定 NPC 愿不愿意袒露私事，后者只描述玩家守不守约。
- 每天开始时，情绪会向平静回落，冲突严重度也会自然下降。
- 未解决冲突会压低靠近和示好的倾向，严重时会阻止送礼、送钱、帮忙等亲近动作。
- 普通冲突仍可靠道歉、礼物、时间逐步修复；严重冲突会进入多步修复链，至少需要“道歉 + 合适礼物 + 时间 + 真正谈开”。
- 多步修复期间，NPC 会从冷硬逐渐软化，而不是被一句客套话立刻重置。
- 严重冲突若被真正修复，关系可能比修复前更深一点，因为这次修复会反过来增加信任和熟悉度。
- 冲突刚恢复后，下一次对话可自然提到“之前的事已经过去了”，而不是像什么都没发生过。

现在又加了一层 **情绪表达风格**，让“同样是不高兴”不再是同一个模板：

- `Flor` 更容易变得安静、谨慎、反思，而不是突然攻击人。
- `Shane` 会更嘴硬、防备，可能直接但不轻易袒露脆弱。
- `Haley` 的不满更明显、更锋利，但小摩擦也更容易过去。
- `Harvey` 更倾向礼貌地担心、迟疑，或用温和边界表达不舒服。
- 防备型角色会更少说原因，但冲突记忆衰减更慢。
- 外露型角色会表现得更明显，但善意回应后也更容易回暖。

这层不只是 prompt 文案：它也会轻微影响每日情绪衰减、冲突衰减、修复响应速度和严重冲突需要的时间。

### 4. 关系与聊天节奏

LivingNPCs 现在会把 **原版好感心数** 和 **自己的长期熟悉度** 一起考虑：

- 关系很浅时，同一天反复聊天会更容易显得礼貌、谨慎、有距离。
- 关系更熟后，同一天多聊几次也可以自然一些。
- 连续多天见面、隔很久没见、今天第一次说话，都会影响下一句对话应该有多热络。
- 这个逻辑会进入 ValleyTalk 的隐藏上下文，避免所有 NPC 都机械地一直热情。
- 关系信任还会决定 NPC 能分享多深的私事：低信任更偏表层寒暄，高信任时才允许更脆弱、更私人的内容自然出现。

AI 对话还会在原版每日对话好感之外，额外给一点“聊天质量奖励”：

- `rapportDelta` 由模型根据聊天效果判断，范围为 `0-30`。
- 普通愉快聊天通常应在 `10-15` 左右。
- 更有温度或更有意义的交流可到 `16-24`。
- 特别投缘时才会接近 `25-30`。
- 每个 NPC 每天通过 AI 对话额外获得的好感有上限，默认是 `30`。

### 4.5. NPC 社会关系层

LivingNPCs 现在还会在 NPC 之间保存一层有限传播的“社区印象”：

- 如果玩家和某个 NPC 连续来往、完成了对方的求助、履行了约定，或在公开场合表现出明显亲密，这件事可能被附近 NPC 看到。
- 亲眼看到的人会保存较高置信度的 `Witnessed` 记忆。
- 系统还会利用 `Data/Characters` 中的 `FriendsAndFamily`，把消息以较低置信度的 `CloseCircle` 形式传给亲近圈层。
- 如果事实本身足够公开，并且发生在 `Town` 或 `Saloon`，少量在场 NPC 还会得到更弱的 `PublicRumor` 印象。
- 社区事实分成 `Public`、`Personal`、`Private` 三档；越私密，越依赖高关系信任或非常小的亲近圈层，不能随便扩散。
- 同一件事进入不同 NPC 的上下文时，还会叠加各自的反应口吻：温和、含蓄、直接、好奇等，所以同一件事不会在每个人嘴里都长得一样。
- ValleyTalk 每次只会拿到当前最相关的一两条社区印象，并被要求对间接消息保持含糊，不把 NPC 写成全知视角。
- 社区消息还有生命周期：`fresh`、`settled`、`fading`，旧消息会降权、衰减，最后自然消失。
- 少量 NPC 会在每天开始时继续向熟人转述新鲜消息；爱表达和好奇的人更愿意传，谨慎或慢热的人更容易把话留在心里。
- 消息经过多次转述后会累计“转述层数”和“失真度”，措辞会逐渐变得更含糊，而不是越传越精确。
- 当前已经有一层稳定圈子：家庭 / 亲近关系、酒吧常客圈、年轻人圈。后两者主要传播公开消息，不会自动扩散私密事。

这样海莉可以隐约知道你最近常和艾米丽说话，罗宾可能听说你帮过玛鲁，而酒吧里当时在场的人也会对你刚做过的事留下模糊印象。

### 4.6. 游戏进程感知

LivingNPCs 现在还会把“这个存档已经走到哪一步”写进隐藏上下文，而不只是看当天季节：

- `路线`：社区中心已修复、Joja 路线，或尚未决定。
- `设施解锁`：巴士、温室、矿车、姜岛、电影院。
- `玩家职业倾向`：根据真实 profession 记录归纳为耕种、钓鱼、采集、采矿、战斗。
- `农场规模`：根据当前作物、完工建筑和动物数量估计为起步 / 小型运转 / 成熟 / 大型经营。
- `家庭阶段`：已婚对象与孩子数量。
- `时间阶段`：第一年春的新来者、第一年安顿中、第二年已融入、第三年及以后老住户。

这层信息会给 ValleyTalk 明确约束：

- 第一年春天不能把玩家写成在镇上生活多年的老熟人。
- 第三年以后也不能还把玩家当成刚搬来的新人。
- 已修好的设施会被当成日常事实，未解锁的设施不该被随口提成已经存在。
- 结婚、生子、农场做大以后，NPC 可以在合适时自然承认这些变化。

这套进度信息现在还分成两层：

- **公开事实**：路线、公共设施、你已经在镇上生活到第几年、婚姻和孩子这类全镇很容易知道的事。
- **私人可知信息**：农场大概规模、玩家更偏向哪类职业路线。只有关系够近、当前就在农场、角色本身因为工作 / 兴趣确实有理由知道，或玩家曾经亲口聊过时，才会进入该 NPC 的上下文。

所以同一个第三年冬天，威利可以自然看出你已经很懂钓鱼，普通路人却不会突然准确报出你的职业倾向；如果你曾亲口告诉弗洛尔自己最近一直在扩建农场，她之后也会比陌生人更知道这件事。罗宾可以大致知道农场已经做大了，但也不会凭空说出你有几格作物、几头动物。

这层进度还会影响主动求助的深度：

- 第一年春天：只允许低负担的一步小忙。
- 第一年其余时间：可以偶尔出现两步请求，但仍以轻量为主。
- 第二年及以后：关系和语境都合适时，才允许更完整的多步骤请求。

现在它还会经过 NPC 自己的价值观再进入对白：

- 社区型角色会把社区中心修复看得更重。
- `Pierre` 和 `Morris` 对社区中心 / Joja 路线的态度会天然分叉。
- `Pam` 会把巴士修复当成和自己更有关的事，`Willy` 会更容易把姜岛看成真实改变，农场型角色会更在意温室，家庭型角色也会更自然地注意婚姻和孩子。
- 这一层同样对 SVE / RSV 角色生效；有手写摘要的角色会更贴脸，没手写摘要的角色会走保守推断。

现在还额外加入了 **SVE / RSV 专属进度感知层**：

| 扩展 | 当前识别的进度 |
| --- | --- |
| SVE | 爷爷的棚屋修复、Apples、魔法林地、Aurora Vineyard 修复、Crimson Badlands、Castle Village Outpost、Susan、Joja Emporium |
| RSV | 到过里奇赛德村、缆车、公共聚会、温室修复、Ridge Forest、Spirit Realm、Ninja House、Undreya、Daia |

这些信息会按 NPC 来源和身份过滤：

- SVE 角色会把已确认的 SVE 里程碑当成自己世界里真实发生的事。
- RSV 角色会把已确认的 RSV 里程碑当成里奇赛德村生活的一部分。
- 原版角色默认只把这些当作远景背景，不会突然像全知视角一样聊扩展剧情。
- 魔法线、冒险线、农场 / 葡萄园线、Joja 线、里奇赛德社区线等角色，会获得更贴身份的个人化提示。
- 未确认的扩展里程碑会被明确标记为未完成，避免 AI 提前剧透或编造玩家已经做过的事。

### 4.7. 调试与评估工具

为了后面越做越大时还能看清“为什么”，现在加了一层调试工具：

| 工具 | 用途 |
| --- | --- |
| `LeftShift + J` | 输出最近 NPC 的状态、记忆、召回结果到 SMAPI 控制台 |
| `livingnpcs_debug [near\|NPC名]` | 输出某个 NPC 当前状态、最近行为选择原因、求助生成适配和记忆召回 |
| `livingnpcs_prompt [near\|NPC名]` | 输出 LivingNPCs 即将注入 ValleyTalk 的完整隐藏上下文 |
| `livingnpcs_export [near\|all\|NPC名]` | 导出 Markdown 调试报告到 `Mods/LivingNPCs/debug_reports/<存档名>/` |
| `livingnpcs_eval` | 在游戏内跑一组轻量运行时诊断，确认关键人格化规则还在 |

导出的报告会包含：

- 每个 NPC 的当前状态可视化表格。
- 最近为什么选择了某个行为。
- 当前为什么适合或不适合提出求助。
- 本次哪些长期记忆、玩家偏好、社区印象被检索进 prompt。
- 完整 ValleyTalk 隐藏上下文预览。

仓库里还新增了一个离线回归检查项目：

```powershell
dotnet run --project LivingNPCs.Diagnostics\LivingNPCs.Diagnostics.csproj -- .
```

它不需要开游戏，主要检查关键调试能力、情绪风格、求助解释、行为选择原因和 README 说明没有被后续改动误删。

### 5. NPC 行为层

当前已实现的安全微行为：

- `FacePlayer`：转向玩家。
- `Emote`：转向玩家并显示表情。
- `ApproachPlayer`：短距离走近玩家。
- `Pause`：短暂停下并注意玩家。
- `LookAround`：环顾四周。
- `StepAway`：后退一步，给关系较浅或互动过密时留一点空间。

行为有两种来源：

- **手动触发**：方便测试，用快捷键直接让附近 NPC 触发一次行为。
- **被动触发**：附近 NPC 在一定概率下自然做出小反应。
- **对话驱动**：AI 对话会留下短期行为倾向，让之后几分钟到几天内的行为更像是从刚才那段交流里长出来的。

行为并不是纯随机：

- 会参考 NPC 自己的轻量性格倾向。
- 会参考当前关系、熟悉度、今天聊天是否太频繁。
- 会参考时间、天气、地点、室内外、附近是否有其他 NPC。
- 会参考世界已经推进到哪一步，避免第三年冬天还像第一年春天那样理解玩家。
- 会参考最近对话是否刚刚让 NPC 更想靠近、保持距离、停下来聊，或把注意力放到某个地点上。
- 会把行为结果重新写回记忆和当前状态，再进入下一次对话上下文。

当前支持的对话驱动行为倾向：

| 倾向 | 作用 |
| --- | --- |
| `companion_walk` | 刚刚说好一起走后，后面真的可能进入短时同行 |
| `visit_location` | NPC 真心提到想去某处，之后若在目标地点相遇，会自然接上这件事 |
| `comforted` / `stay_near` / `pause_to_talk` | 玩家安慰或拉近关系后，更愿意靠近、停留和回应 |
| `offended` / `give_space` | 被冒犯或需要空间后，更可能疏远或后退 |

### 6. AI 对游戏世界的受控影响

现在 AI 已经可以“提出”少量世界动作请求，但 **不能直接随便改游戏**。流程是：

```text
AI 在隐藏元数据里提出动作请求
-> LivingNPCs 校验关系、地点、冷却、频率和上限
-> 只有通过白名单和安全检查后才执行
-> 执行结果写回 NPC 记忆
```

当前白名单动作：

| 动作 | 当前规则 |
| --- | --- |
| `give_small_gift` | 关系至少有一点熟悉；每天每个 NPC 最多一次 AI 礼物 |
| `give_meaningful_gift` | 至少友好；并且还要满足高关系、近期特殊事件、或重要长期记忆触发之一；有独立冷却 |
| `give_money` | 至少友好；默认 100g，受配置上限约束 |
| `water_nearby_crops` | 至少友好；只能在农场；只给附近已种植但未浇水的作物浇水 |
| `walk_together` | 至少有一点熟悉；只做短时间临时同行，不改 NPC 原始日程表 |
| `escort_to_location` | 只做有时间限制、路径受限的短时陪走；不会为了跨地图改日程或瞬移 |
| `festival_interaction` | 只在节日 / 事件场景中允许轻量特殊互动 |
| `assist_quest` | 只围绕玩家已有任务做轻量协助，不直接完成任务 |

明确暂未开放的危险动作：

- 种植作物。
- 传送 NPC。
- 永久修改日程。
- 改任务、改剧情、改世界状态。
- 任何不在白名单里的任意指令。

### 7. 礼物系统

AI 礼物目前分成两层：

#### 小礼物

用途：日常、轻量、低价值。

会根据以下因素打分：

- NPC 的性格标签。
- 当前原版好感 / LivingNPCs 关系层级。
- 当前季节。
- NPC 的职业和背景。
- 这次对话里刚聊到的内容。
- 长期记忆里反复出现的重要主题。
- 玩家偏好记忆，权重高于普通主题记忆。

例如：

- 聊到困、熬夜、学习，咖啡会更容易入选。
- 聊到花、花园，花类物品会更容易入选。
- 如果 NPC 已经记得玩家喜欢花、矿物或某类食物，相关礼物会明显优先。
- 如果 NPC 记得玩家讨厌某种物品或类别，相关候选会被强力压低。
- 角色本身偏学术、偏务实、偏自然，候选池会受影响。

#### 有意义的礼物

用途：不是更频繁，而是更郑重。

只有在以下情形才允许触发：

- 关系足够深；
- 或刚经历过特殊事件；
- 或当前对话明显碰到了重要长期记忆。

当前候选会偏向宝石、花、甜点等更有“心意”的物品，并且有独立冷却天数，默认同一 NPC 至少相隔 7 天才能再送一次。

### 8. NPC 主动求助

LivingNPCs 现在有一层轻量“求助任务”系统，先独立于原版任务栏运行，但已经有自己的游戏内列表。

当前支持两类：

| 类型 | 当前完成方式 |
| --- | --- |
| `item_request` | NPC 请求一个低价值、白名单内的物品；玩家把对应物品送给该 NPC 后自动完成 |
| `question_request` | NPC 向玩家请教一个问题；后续 AI 对话会判断玩家是否给出了有用回答 |

生命周期现在分得更细：

- 新请求先是 `Offered`：NPC 只是开口问了，玩家还没答应，所以不会立刻进入任务栏。
- 玩家明确答应后，AI 回传 `accepted`，请求进入 `Pending`，才会投影到原版任务日志。
- 玩家婉拒时，AI 回传 `declined`，不会生成任务，只留下轻量关系记忆。
- 请求可以有最多 3 个步骤；任务栏只显示当前步骤，完成中间步骤会推进，最后一步完成才给完整奖励。
- 若完成后的 `followUpPotential` 指向 `new_commitment`、`shared_activity` 或 `deeper_relationship`，这次帮助会更容易在后续对话里发展成新约定或更深关系。
- 失败反应会区分“没答应所以没做”和“答应了但没做”，并根据角色性格变成直接、退缩或温和的不同表现。

触发前会先过几道门：

- 同一个 NPC 不能已经有待完成求助。
- 默认需要至少一点熟悉度，并且关系信任达到配置阈值。
- 有明显未解决冲突时不会开口求助。
- 当前正处于 `Angry` / `Upset` 时不会求助。
- 同一 NPC 两次新求助之间有冷却天数。
- 生成时还会参考 NPC 的职业、性格、当前季节，以及眼下这轮对话里刚聊到的内容；例如学者型角色更容易请教问题，务实角色更容易提出贴近工作的小忙。

查看方式：

- NPC 求助现在会直接显示在原版任务日志里。
- 条目会显示是谁在求助、需要什么 / 想问什么、还有多久到期。
- LivingNPCs 仍然自己维护底层状态，只是把它投影进原版任务栏，手感会更像原版任务扩展。

物品请求会额外考虑游戏进度：

- 只会从**当前季节**能自然获得的低价值季节物品里挑选。
- 矿物类请求会等到矿井开放后才可能出现。
- 紫水晶这类更靠后的矿物，会等到玩家确实下到足够深的矿层后才进入候选。
- 如果当前进度下没有合适物品，AI 会更倾向于提出一个问题型求助，或干脆先不求助。

完成后：

- NPC 会进入轻度 `Grateful`。
- 关系信任、熟悉度和履约信任都会提高。
- 额外增加 `50-100` 点好感，具体数值会固定落在配置范围内。
- 默认会给一份小谢礼。
- 隔天可能主动补一句后续感谢。
- 这次帮助会写入“共同经历”里程碑，让之后更容易自然长出新的约定。

拒绝或过期时：

- 只产生很轻的 `Disappointed`，不会把普通小忙当成严重失约。
- 这层设计故意比正式约定更轻，方便以后继续扩展成更完整的任务系统。

### 9. ValleyTalk 上下文桥接

LivingNPCs 会把一份“隐藏连续性摘要”推给 ValleyTalk，而不是只塞几条原始日志。

当前摘要会告诉模型：

- NPC 当前应该偏什么语气。
- 与玩家现在熟到什么程度。
- 今天是不是已经聊得偏多。
- 最近是否送过礼物。
- 是否发生过事件互动。
- 附近有没有其他 NPC。
- 当前时间、天气、地点会不会让 NPC 更忙、更安静、更公开场合化。
- 最近的重要长期记忆是什么。
- NPC 已知的玩家偏好是什么。
- 这个 NPC 近期亲眼看到或听说过哪些玩家与他人的来往。
- 双方是否还有待履行或刚失约的约定。
- NPC 是否已经向玩家提出过尚未完成的小忙。
- 当前是否存在未解决冲突，或者是否刚刚和解。
- 玩家是否有已被该 NPC 接受的称呼。

同时摘要会要求模型：

- 不要直接复述这些机制。
- 不要提到 LivingNPCs、prompt、AI、JSON。
- 只自然吸收最 relevant 的一两点，写成正常 NPC 台词。

### 10. 自定义 NPC / SVE / RSV 支持

目前两层兼容都已经做了：

#### LivingNPCs

- 优先使用内置角色资料。
- SVE 当前内置识别到的角色已经补齐手写摘要；常见内部别名也会落到同一份资料上。
- RSV 已补充一批主要村民的手写摘要；其余次要角色仍会继续使用 `Data/Characters` 推断兜底。
- 对没有手写资料的自定义 NPC，会从游戏加载后的 `Data/Characters` 推断基础性格。
- 已给 Stardew Valley Expanded 和 Ridgeside Village 做了专属识别：
  - 有摘要资料的角色会带上更明确的背景和性格提示。
  - 没有手写摘要的角色，也会带上所属 Mod 的世界观提示并回退到基础推断。
- 世界进度感知、NPC 个性化态度、关系记忆、求助系统这些 LivingNPCs 层能力也会应用到 SVE / RSV 角色。
- 目前单独建模的仍主要是**原版公共世界进度**；SVE / RSV 自己额外增加的专属剧情阶段，后续还可以再单独补一层。
- 新增 `LivingNPCs/npc_profiles/` 外部人物资料目录：
  - 支持社区用 JSON 直接补充或覆盖角色资料。
  - 支持单角色文件、数组文件、以及 `{ "profiles": [...] }` 三种格式。
  - 内置了 `_template.json` 和中文说明，后续接受社区提交时不必改 C#。

#### ValleyTalk fork

- 能读取游戏运行时已经加载的自定义 NPC。
- 增加 `AllowLocalContentPackDialogueForAi`，默认允许本地内容包对白进入 AI 上下文。
- 因此在当前本地 fork 中，SVE / RSV 角色也可以触发 AI 对话。
- 若某个自定义 NPC 没有现成 biography，fork 会从 `Data/Characters` 自动生成轻量 biography，至少包含年龄、礼貌程度、社交倾向、乐观程度、是否可恋爱、家庭 / 朋友关系等基础信息。

### 10.5. Prompt 体积与资料占比

下面这些数字是按“字符数”估算的，不是严格 token 数。不过它们和 SMAPI 日志里的 `promptChars` 更接近，可以用来判断为什么某些 NPC 对话会更慢。

#### 固定世界背景

| 内容 | 大致字符数 | 说明 |
| --- | ---: | --- |
| 原版游戏世界摘要 | 约 13,675 | ValleyTalk 的基础世界设定，很多对话都会带上 |
| SVE 世界补充 + 地点补充 | 约 4,950 | 安装 SVE 扩展上下文时会额外加入 |
| RSV 世界补充 | 暂无同级 biography/world-summary 包 | 当前主要依赖角色短人设、内容包对白和运行时资料 |

这部分是 prompt 里的大头之一。尤其是 SVE 角色，通常会同时吃到原版世界摘要和 SVE 世界补充。

#### ValleyTalk 角色 Biography

SVE 的 ValleyTalk biography 文件当前有 23 个，合计约 68,947 字符，平均每个角色约 2,998 字符。

| SVE 角色示例 | 大致字符数 |
| --- | ---: |
| Claire | 约 2,503 |
| Sophia | 约 2,530 |
| Martin | 约 2,848 |
| Lance | 约 3,463 |
| Henchman | 约 4,636 |

注意：这些 biography 不是一次性全部塞进 prompt，而是当前正在对话的 NPC 使用自己的那一份。

RSV 目前没有同等体量的 ValleyTalk biography 文件。RSV 角色更多依赖 LivingNPCs 手写短人设、游戏运行时的 `Data/Characters` 推断，以及 RSV 内容包里可读取的对白样例。

#### LivingNPCs 手写人设

LivingNPCs 自己写的人设是轻量补充，不是大段传记。它主要告诉 AI 这个角色的气质、背景和对白方向。

| 来源 | 条目数 | 总字符数 | 平均每人 |
| --- | ---: | ---: | ---: |
| SVE 手写短人设 | 28 | 约 9,319 | 约 333 |
| RSV 手写短人设 | 42 | 约 12,757 | 约 304 |

进入实际 prompt 时，短人设通常会被写成几行：

- `Character profile source`
- `Disposition`
- `Character background`
- `Character dialogue cue`

再加上当前关系姿态后，一个角色大多约 350-450 字符。也就是说，LivingNPCs 这部分不是主要耗时来源，它更像精确调味。

#### 角色样例对白

ValleyTalk 当前最多会选 20 条样例对白放进 prompt，用来模仿角色语气。实际选中哪 20 条会受季节、好感、日期、上下文排序影响，下面只是按本地内容包粗略量的参考值。

| 角色 | 内容包总对白数 | 全部字符数 | 前 20 条字符数 |
| --- | ---: | ---: | ---: |
| Claire | 126 | 约 3,795 | 约 596 |
| Sophia | 186 | 约 7,224 | 约 1,084 |
| Martin | 50 | 约 1,592 | 约 638 |
| Flor | 80 | 约 4,201 | 约 893 |
| Keahi | 64 | 约 2,805 | 约 706 |
| Jio | 94 | 约 3,350 | 约 354 |

所以样例对白通常不是最大的那块，但也会稳定贡献约 400-1,200 字符。有些角色对白更碎、更短，有些角色对白更长，差异会比较明显。

#### 为什么 prompt 会到 20k-35k

一次 SVE / RSV AI 对话的 prompt 大致由这些部分叠起来：

```text
原版世界摘要
+ SVE / 扩展世界摘要
+ 当前 NPC biography 或自动 biography
+ 当前 NPC 样例对白
+ 当前游戏状态
+ 当前聊天历史
+ ValleyTalk 事件历史
+ LivingNPCs 隐藏上下文
+ 长期记忆 / 玩家偏好 / 社区印象 / 约定 / 求助 / 情绪冲突
```

因此一个 SVE NPC 例如 Claire，可能大致是：

```text
原版世界摘要约 13.7k
+ SVE 世界补充约 5.0k
+ Claire biography 约 2.5k
+ 样例对白约 0.6k-1.2k
+ LivingNPCs 人设约 0.4k
+ 其他状态、历史、记忆、当前对话
= 很容易到 20k 以上
```

如果要继续优化速度，优先压缩的顺序应该是：

1. 压缩或缓存 ValleyTalk biography。
2. 把样例对白数量从 20 条做成可配置，例如 8 / 12 / 20。
3. 控制跨会话事件历史条数。
4. 给固定世界摘要做缓存或分层摘要。
5. 最后才考虑压缩 LivingNPCs 手写人设，因为它本来就很小。

### 11. 聊天记录与可观察性

ValleyTalk 当前会为每个存档、每个 NPC 导出可读聊天记录：

```text
D:\SteamLibrary\steamapps\common\Stardew Valley\Mods\ValleyTalk\conversation_logs\<存档文件夹>\<NPC名>.md
```

日志不是纯粹的一句一句对白，而是按聊天轮次保留：

- 玩家输入。
- NPC 回复。
- 游戏日期和时间。

LivingNPCs 还提供“查看记忆快捷键”，可以在 SMAPI 控制台输出附近 NPC 当前状态、互动节奏、长期记忆和最近行为，方便调试。

### 12. 长期行为与约定

NPC 现在可以把真正达成的未来计划单独记下来，而不是只把它们当成一句普通 `promise` 文本。

当前约定字段包括：

- 类型：`meet_again`、`go_together`、`help_task`、`celebrate_together`、`share_activity`
- 时间：当前日期加 `dueInDays` 后得到的绝对游戏日，以及约定时刻
- 地点：标准地点键和自然语言地点名
- 状态：`Pending`、`Waiting`、`Fulfilled`、`Expired`

当前触发逻辑：

- 到约定时间后，如果玩家已经在约定地点，NPC 会临时来到玩家附近并完成约定。
- 如果玩家还没到，NPC 会先去约定地点的安全会面点等待。
- 玩家在宽限时间内到达约定地点后，NPC 会主动靠近并用头顶气泡打招呼。
- 约定地点不是无限大的模糊区域；例如“农场”会优先选择农舍门口附近，兼容特殊农场时会优先找农舍入口 warp 附近。
- 约定前一天，如果再次聊天，NPC 会自然提醒一次第二天的安排。
- 约定当天清晨，如果双方恰好在附近，NPC 有一定概率用头顶气泡提醒今天还有安排。
- 如果 NPC 已在约定地点而玩家未到，状态会进入“等待约定”的短期上下文。
- 超过配置的宽限时间后，约定会转为 `Expired`，NPC 会尽量回到当前日程位置附近。
- 下次正常开始对话时，过期约定会作为高优先级上下文送入 ValleyTalk，鼓励 NPC 自然提起这次失约。
- 如果约定真的被完成，NPC 会得到一点额外的熟悉度 / 开放度反馈，并把这次经历写成单独的“共同经历”里程碑。
- 共同经历会在几天后长出一次自然后续，例如 NPC 主动提到“上次一起去那里挺开心”，也会让它更愿意在合适场景下提出相似的新约定。
- 失约不再只是一次提示：会降低“履约信任”，连续失约会造成更明显的失望和 `promise` 冲突；玩家之后若真诚道歉并给出新的具体约定，可以开始修复。

这套系统当前仍然刻意保守：

- 不会改写永久日程。
- 只会为明确约定做一次临时会面移动，完成、失约或离开窗口后仍尽量回到原本日程。
- `escort_to_location` 目前是“陪你先走一段”的安全实现，受同行时长和当前地图路径限制约束。

### 13. Token 消耗统计

ValleyTalk 现在会按“当前会话”和“当前存档”累计 AI 请求的 token 消耗：

- 支持优先读取提供商返回的官方 usage：OpenAI 兼容接口、Claude、Gemini、VolcEngine、llama.cpp。
- 如果供应商没有返回 usage，或当前是流式输出，则会写入本地估算值，并明确标成 `estimated`。
- 汇总维度包括：总量、按模型、按 NPC、最近请求。
- 当前存档的统计会随存档保存，并自动导出到 `ValleyTalk/token_usage/<存档名>.md`。

在 SMAPI 控制台可用：

- `valleytalk_tokens`：查看当前会话和当前存档汇总。
- `valleytalk_tokens export`：立即导出当前存档统计。
- `valleytalk_tokens reset`：重置当前存档统计。

## 当前整体流程

```text
玩家与 NPC 互动
-> LivingNPCs 记录开始对话 / 礼物 / 事件上下文
-> LivingNPCs 生成隐藏上下文摘要并推给 ValleyTalk
-> ValleyTalk 结合角色资料、游戏状态、历史对话和 LivingNPCs 摘要生成回复
-> ValleyTalk 同时输出隐藏元数据
-> LivingNPCs 解析元数据
-> 更新长期记忆、玩家偏好记忆、对话驱动行为、长期约定、情绪 / 冲突、额外好感、后续头顶气泡、受控世界动作
-> 在目击者与亲近圈层中写入有限传播的社区印象
-> 新状态进入下一轮对话
```
